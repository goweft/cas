# Conversational Agent Shell (CAS)
## A New Shell Paradigm for Heddle

**Version:** 0.1.0-concept  
**Author:** Steve Gonzalez (@gostev)  
**Date:** March 2026

---

## Definition

**Conversational Agent Shell (CAS)** is a session-layer interface paradigm where the primary user interaction surface is a persistent natural-language conversation. Unlike CLI or GUI shells, CAS does not present a static spatial interface. Instead, application workspaces are dynamically generated by specialized agents in response to user intent expressed through conversation. Workspaces are rendered alongside the conversation as interactive, editable panels managed by an underlying agent mesh (Heddle) that handles lifecycle, containment, resource allocation, and inter-workspace communication.

## The Problem

Current operating systems force users to navigate a mismatch between intent and interface. "I need to analyze this quarter's revenue" becomes a chain of application-specific actions: open spreadsheet → find file → import data → select columns → create chart. Each step requires domain knowledge about the application's interface.

Existing AI approaches make this worse, not better:
- **AI overlays** (Microsoft Copilot) bolt AI onto existing tools — users reject this as intrusive and unhelpful
- **AI automation** (OpenClaw, Claude Computer Use) replace user agency — the AI operates your tools for you
- **AI-only** (YOLO Linux) eliminate the GUI entirely — users lose spatial awareness and direct manipulation

None generate purpose-built tools from conversation while preserving user agency.

## The Solution

CAS proposes a hybrid: **conversation for intent expression, workspaces for direct manipulation.**

The user describes what they need. An agent generates a purpose-built workspace — a document editor, data viewer, terminal, or communication panel — that appears alongside the chat. The user operates the workspace directly. The conversation continues alongside, available for further requests, modifications, or orchestration across workspaces.

The agent builds the tool. The user wields the tool. Neither role is subordinated.

## Theoretical Foundation

CAS synthesizes three decades of HCI research:

- **Norman's Two Gulfs (1986):** Conversation narrows the Gulf of Execution (easier to express intent). Workspaces narrow the Gulf of Evaluation (easier to perceive state). The hybrid addresses both simultaneously.

- **Shneiderman vs. Maes (1997):** The foundational HCI debate — direct manipulation vs. interface agents. CAS resolves it: agents handle workspace generation (Maes), users retain direct manipulation of generated workspaces (Shneiderman).

- **Horvitz's Mixed-Initiative Principles (1999):** Decision-theoretic framework for when AI should act autonomously vs. ask vs. stay silent. CAS's Conductor uses this to determine when to suggest workspaces.

## Security Model: Deterministic Agent Contracts

CAS addresses the fundamental security problem of agent-mediated computing: the probabilistic Trusted Computing Base.

**The problem:** LLMs cannot guarantee deterministic behavior. A model that correctly refuses a malicious instruction 99% of the time still permits it 1% of the time. You cannot build provable security on probabilistic foundations.

**The solution:** Wrap every agent in a **deterministic contract** — preconditions, postconditions, and invariants enforced by non-LLM code that the agent cannot modify, bypass, or reason about. Based on Bertrand Meyer's Design by Contract (1986) and Dennis & Van Horn's capability-based security (1966).

```yaml
contract:
  preconditions:
    - user_session_active: true
    - operation_in_scope: true
  invariants:
    - filesystem_scope: "/tmp/workspace-{id}/"
    - network_access: false
    - cannot_modify_own_contract: true
    - audit_log_append_only: true
  postconditions:
    - no_files_modified_outside_scope: true
    - audit_entry_created: true
  on_violation: quarantine
```

Contracts are loaded from signed YAML at agent startup, validated by Heddle's existing signing module, and enforced by deterministic Python code external to the agent. The agent cannot modify its own contract. Violations trigger immediate quarantine and audit logging.

## Architecture

CAS is a module within Heddle, not a separate project.

```
heddle/
├── cas/              # Conversational Agent Shell
│   ├── shell.py      # Chat session, workspace lifecycle
│   ├── workspaces.py # Workspace types and generation
│   ├── contracts.py  # Deterministic contract enforcement
│   ├── renderer.py   # Workspace HTML generation
│   └── conductor.py  # Behavioral learning (Phase 2+)
├── security/         # Existing — CAS inherits all of it
├── mcp/              # Existing — agent communication
├── runtime/          # Existing — agent lifecycle
├── config/           # Existing — YAML schemas
└── web/              # Existing — extend with CAS routes
```

## Roadmap

**Phase 1 — Proof of Concept:** Chat + one workspace type (document). Contracts. Split-panel UI.  
**Phase 2 — Workspace Expansion:** Data, terminal, and IDE workspaces. Sandboxed execution.  
**Phase 3 — Communication:** Unified messaging (Discord/Telegram/Signal). Encryption agent.  
**Phase 4 — Conductor:** Behavioral learning. Proactive workspace suggestions. Task-aware lifecycle.  
**Phase 5 — Session Manager:** CAS as login shell. Wayland compositor integration. Full desktop replacement.
